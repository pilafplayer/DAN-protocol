## Abstract

DAN is a unidirectional lightweight asynchronous serial communication protocol designed for embedded systems requiring minimal hardware resources. The protocol operates over a single data line and a common ground reference, using a synchronization sequence embedded within each frame to recover the transmitter's bit period. 

The protocol is intended to minimize implementation complexity while eliminating the need for preconfigured communication parameters such as baud rate, payload length and parity configuration.

# 1. Introduction

DAN is an asynchronous serial communication protocol designed for embedded systems where simplicity and minimal hardware requirements are priorities.

Unlike conventional asynchronous serial links that require transmitter and receiver to share communication parameters (e.g. UART), DAN does not rely on a predefined baud rate, predefined word size and a predefined parity enabling shared between transmitter and receiver.

Instead, each frame contains a synchronization pattern that allows the receiver to determine the transmitter bit period dynamically. 

The protocol uses a single data line and a common ground reference, enabling communication with only two electrical connections.

# 2. Design Goals

The main design goals of DAN are:

- Minimal physical wiring requirements.
- Simple implementation and no shared presets required.
- No requirement for accurate shared clock sources.
- Support for small embedded systems.
- Deterministic frame structure.
- Simple error detection.

# 3. Physical Layer

DAN uses a two-wire physical interface:

| Signal | Description |
|---|---|
| DATA | Unidirectional serial data line |
| GND | Common reference |

The DATA line remains HIGH during idle state.

Logical levels:

| State | Voltage |
|---|---|
| LOW | GND |
| HIGH | VCC |

# 4. Frame Format

| Field | Size | Description |
|---|---|---|
| Start | 1 bit | Frame start indicator |
| Synchronization Pattern | 3 bits | Receiver alignment sequence |
| Word Size | 4 bits | Payload length |
| Word | 1-16 bits | Data |
| Parity Enable | 1 bit |  Enabling the parity checking |
| Parity | 1 bit | Parity checking |
| Stop | 1 bit | Frame termination |

# 5. Field Definitions

## 5.1 Start Field

The Start field indicates the beginning of a DAN frame transmission.

During the idle state, the DATA line shall remain at logical HIGH.

A frame transmission begins when the DATA line transitions from HIGH to LOW. This falling edge indicates the beginning of the Start field.

The Start field consists of a single logical `0` bit.

The Start field duration is equal to one transmitter bit period:

$$
T_{start}=T_{bit}
$$

The receiver shall detect the Start field using the falling edge transition. The Start field shall not be sampled, since the receiver has not yet determined the transmitter bit period.

After the completion of the Start field, the transmitter shall immediately transmit the Synchronization Pattern field.

The receiver shall use the Synchronization Pattern to determine the transmitter bit period before sampling any subsequent fields.

## 5.2 Synchronization Pattern

The Synchronization Pattern allows the receiver to determine the transmitter bit period before decoding the remaining frame fields.

The field consists of a fixed 3-bit 101 sequence:

![Timing Diagram](../diagrams/timing-diagram.svg)


The transmitter shall maintain a constant bit period during the entire frame transmission.

The synchronization process is performed independently for each received frame. The receiver shall not retain the calculated bit period for future communications.

### Bit Period Measurement

The receiver shall determine the bit period by measuring the time interval between two consecutive rising edges generated by the Synchronization Pattern.

For the sequence `101`, the rising edges occur at the beginning of the first and third bits of the field.

The measured interval is defined as:

$$
T_{sync}=T_{R2}-T_{R1}
$$

where:

- T<sub>R1</sub> is the timestamp of the first rising edge.
- T<sub>R2</sub> is the timestamp of the second rising edge.
- T<sub>sync</sub> represents two bit periods.

The transmitter bit period shall be calculated as:

$$
T_{bit}=\frac{T_{sync}}{2}
$$

where:

- T<sub>bit</sub> is the estimated duration of one transmitted bit.

### Receiver Sampling Reference

The second rising edge of the Synchronization Pattern marks the beginning of the third bit of the field.

After detecting this edge, the receiver shall wait:

$$
\frac{T_{bit}}{2}
$$

before sampling the third synchronization bit.

The third synchronization bit shall always contain the logical value `1`.

After sampling the third synchronization bit, the receiver shall wait one complete bit period:

$$
T_{bit}
$$

before sampling the first bit of the Word Size field.

The sampling procedure for the remaining fields is defined in Section 6 (Receiver Operation).

### Synchronization Fault

The Synchronization Pattern is not used as a frame identifier. Its purpose is exclusively clock recovery.

However, the third synchronization bit is a mandatory fixed value.

If the sampled third synchronization bit is different from logical `1`, the receiver shall mark the frame as faulty and discard all subsequent data fields from that frame.

A synchronization fault shall be handled in the same manner as a parity verification failure.


## 5.3 Word Size 

The Word Size field defines the number of bits contained in the Word field.

The field has a fixed length of 4 bits and is transmitted immediately after the Synchronization Pattern field.

The Word Size field uses a linear binary encoding, where: 

$$
Word_{Length}=Word_{Size} + 1
$$

This encoding permits all sixteen possible 4-bit values to represent valid payload lengths, as 0000 can't represent a payload with 0 as its length.
The encoding is defined as:

| Word Size Field | Word Length |
|---|---|
|0000|1 bit|
|0001|2 bits|
|0010|3 bits|
|0011|4 bits|
|0100|5 bits|
|0101|6 bits|
|0110|7 bits|
|0111|8 bits|
|1000|9 bits|
|1001|10 bits|
|1010|11 bits|
|1011|12 bits|
|1100|13 bits|
|1101|14 bits|
|1110|15 bits|
|1111|16 bits|

All 4-bit combinations are valid Word Size values.

The Word Size field shall be transmitted most significant bit (MSB) first (i.e. left-to-right).

The receiver shall sample each bit of the Word Size field at the center of its bit period.

The bit period used for sampling is obtained from the Synchronization Pattern field.

After the synchronization process is completed, all bits within the frame shall be sampled at the center of their respective bit periods.

The Word Size field defines only the length of the Word field.

The following fields are independent of the Word Size value:  

- Parity field
- Stop field

The Word Size field is not included in parity calculation.

Parity calculation is performed exclusively over the bits contained in the Word field.

## 5.4 Word Field

The Word field contains the payload data transmitted by the DAN frame.

The size of the Word field is defined by the value contained in the Word Size field.

The Word field supports payload sizes from 1 bit to 16 bits.

The transmitter shall transmit only the number of bits specified by the Word Size field. The Word field shall not be padded with additional bits.

The Word field shall be transmitted most significant bit first (MSB first).

For example, an 8-bit Word with the value:

Word = 01010110<sub>2</sub>

![Word Diagram](../diagrams/word-diagram.svg)

0 → 1 → 0 → 1 → 0 → 1 → 1 → 0 


All possible bit combinations are valid Word field values.

DAN does not define the interpretation of the Word field contents.

The Word field is treated as raw binary data. The meaning of the transmitted bits, such as numerical values, commands, addresses, or other data structures, is defined exclusively by the application layer.

### Parity Calculation

When parity is enabled, parity calculation shall be performed exclusively over the bits contained in the Word field.

The parity calculation does not include:

- Start field
- Synchronization Pattern field
- Word Size field
- Parity Enable field
- Stop field

The parity bit is calculated based only on the number of logical HIGH (`1`) bits contained in the Word field.

The parity calculation method is defined the next Section.

## 5.5 Parity

The DAN protocol uses optional even parity for error detection.

The parity field consists of two bits:

| Bit | Description |
|---|---|
| P_EN | Enables parity verification |
| P | Even parity bit |

When Parity Enable (P_EN) is set to 1, the transmitter calculates the parity bit based on the Word field.

The parity bit is selected so that the total number of logical HIGH bits in the Word field plus the parity bit is even.

When P_EN is set to 0, the parity bit shall be ignored by the receiver.

## 5.6 Stop

The Stop field indicates the end of a DAN frame transmission.

The Stop field consists of a single logical HIGH (`1`) bit transmitted after the Parity field.

The Stop field duration shall be equal to one transmitter bit period:

$$
T_{stop}=T_{bit}
$$

During the Stop field, the DATA line shall remain at logical HIGH.

The receiver shall sample the Stop field at the center of its bit period.

A valid frame termination is detected when the sampled Stop bit is logical HIGH (`1`).

If the sampled Stop bit is logical LOW (`0`), the receiver shall mark the frame as invalid and discard the received frame.

A Stop field error shall be treated as a frame termination error.

After a valid Stop field has been detected, the transmitter may keep the DATA line at logical HIGH, which corresponds to the Idle state.

The receiver shall return to the Idle state after processing the Stop field.

# 6. Receiver Operation

## 6.1 General Receiver Requirements

A DAN receiver shall continuously monitor the DATA line while the bus is in the idle condition.

Whenever a valid Start condition is detected, the receiver shall initiate a new frame reception.

Each received frame shall be processed independently. The receiver shall not retain synchronization parameters, timing information, or any other frame-specific information after frame reception has completed. The receiver shall not keep the clock timins and other informations in order to allow the transmitter to change any of these parameters at any given time, without having to specify again to the receiver on what conditions it is working at.

For every received frame, the receiver shall perform the following operations in the specified order:

1. Detect the Start field.
2. Recover the transmitter bit period from the Synchronization Pattern.
3. Receive and decode the Word Size field.
4. Receive the Word field.
5. If parity verification is enabled, verify the received parity.
6. Verify the Stop field.
7. Complete frame reception.

The receiver shall treat every received frame as one of the following:

- Valid;
- Valid with parity error;
- Invalid.

A frame shall be considered **Valid** when all mandatory fields are successfully received and verified.

A frame shall be considered **Valid with parity error (PE)** when all mandatory fields are successfully received and verified, but parity verification fails.

A frame shall be considered **Invalid** whenever any mandatory protocol requirement is violated, including but not limited to:

- Invalid Synchronization Pattern;
- Invalid Stop field;
- Any other protocol violation defined by this specification.

Invalid frames shall be discarded.

Frames containing parity errors shall not be discarded solely because of the parity error. The received Word shall remain available together with an indication that parity verification failed.

Upon completion of frame reception, regardless of the reception result, the receiver shall return to the idle condition and wait for the next valid Start condition.

## 6.2 Start Detection

The receiver shall continuously monitor the DATA line while in the idle condition.

A Start condition shall be detected upon the occurrence of a HIGH-to-LOW transition on the DATA line, as defined in Section 5.1.

Upon detecting a valid Start condition, the receiver shall immediately begin the frame reception procedure.

A frame shall not be considered valid solely because a Start condition has been detected. Frame validity shall only be established after successful synchronization and verification of the subsequent mandatory fields (Frame Validity will be covered in Section 9).

Immediately following the Start field, the receiver shall begin the synchronization procedure defined in Section 6.3.
